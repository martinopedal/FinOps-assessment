- **2026-05-13** , Issue #44 completed (PR #46): Fixed `tests/test_live_collectors.py` failing on fresh `[dev]` install. **Root cause:** `[dev]` extras install `types-requests` (mypy stubs) but not `requests` runtime; `[live]` extras install `requests>=2.32`. The 6 live-collector tests lazily import collector modules that depend on `requests`, causing `ModuleNotFoundError` at test runtime when `[live]` not installed. CI was green because `.github/workflows/ci.yml:34` installs `[dev,live]`. **Solution (Option A):** Added `pytest.importorskip("requests", reason="Live collectors require [live] extras: pip install -e .[live]")` at module level in `tests/test_live_collectors.py` before any imports. Pattern: `pytest.importorskip(module_name)` is pytest's canonical guard for optional-dependency tests , checks importability at module load time, skips entire module with clear message if unavailable. **Verification harness:** (1) Parent venv with `[live]`: `pytest tests/test_live_collectors.py -v` → 6 PASSED; (2) Fresh `[dev]`-only venv: `python -m venv .venv-dev-only && pip install -e .[dev]` → same pytest invocation → 1 module SKIPPED; (3) Full suite: 189 passed, 3 skipped (no regressions). **Learnings:** The `[dev]`/`[live]` extras boundary is intentional , dev tooling (pytest, mypy, ruff, type stubs) vs production deployment dependencies (azure-identity, requests). Type stubs allow type-checking collector code without installing runtime deps. When tests exercise optional-dependency code paths, guard with `importorskip` at module level rather than conditional imports or merging extras. Fresh-venv verification is critical for extras-boundary PRs , parent venv accumulates dependencies across sessions and hides missing-dep failures. **Scratch-file hygiene:** Added `pr-body.md` and `stage-*.md` patterns to `.gitignore` per Noor's #42 lockout lesson , §11 ceremony artefacts belong in PR body, not repo root. **CI expectation:** No change; CI already installs `[dev,live]` so `importorskip` is a no-op in CI matrix.
