- **2026-05-12** , PR #42 round-2 lockout-revision (Diego): Python's `\b` word boundary treats underscore as a word character, so regex patterns like `\bpurchase\b` silently fail against snake_case Literal values like `"purchase_recommended"` or `"exchange_recommended"` , the exact surface the reflective Literal walk was designed to protect. Fixed by replacing `\b` anchors with explicit alphabetic lookaround `(?<![a-zA-Z])(purchase|buy|exchange|modify)(?![a-zA-Z])` (case-insensitive). This catches `exchange_recommended`, `purchase_now`, `BUY_SP` as intended, while still excluding "purchased" (past tense). Added three positive-control assertions and one negative control to prove the fix. This regex gotcha applies to #30 (agreement-types) which will also use Literal enums: use explicit lookaround for token-boundary scanning of snake_case strings.
- **2026-05-12** , PR #30 (issue #30): Agreement-multiplier pricing-profile contract. Third extension of pricing.py observation pattern. Key design choices: (1) Multiplier bounds [0.0, 1.0] , multipliers >1.0 (above list, e.g., sovereign-cloud markup) are out of scope for v1; model them as list-price observations with multiplier=1.0 and document the premium separately. (2) Currency posture: require currency match between profile and observation; rules reject mismatches. No auto-conversion (exchange rates are temporal). USD baseline for v1; future multi-currency expands both contracts together. (3) Scope-vs-product: profiles are scoped (`Literal["azure", "m365", "github", "ado"]`); customers with multiple agreements (EA for Azure, CSP for M365) supply multiple profiles. Rules filter by scope before applying. (4) Effective-date validation: rules are responsible for checking `observation.observed_at` falls within `[effective_from, effective_to]`; the model does not enforce this (observations and profiles are independent inputs). (5) docs/plan.md §10 open-question 1 RESOLVED: pricing profiles are customer-supplied inputs, NOT catalogue constants; default posture is list price (multiplier=1.0, source=default_list). Complete pricing.py surface now has 3 model families (region-price, commitment-coverage, pricing-profile), all sharing pydantic idioms: `extra="forbid"`, ISO 8601 date constraints, `source` provenance fields, `currency: Literal["USD"]` baseline, no hardcoded tenant data. Natural next extension: commitment-discount overlap with agreement-multiplier (rule layer figures out interaction order, not the contract); multi-currency support (expand `currency` in both observations and profiles simultaneously, rules handle conversion); sovereign-cloud regions (add `cloud` scope after security review).
