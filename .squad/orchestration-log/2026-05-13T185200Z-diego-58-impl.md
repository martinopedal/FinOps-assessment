## Notable Implementation Decision

**Advisory finding key collision fix (embedded in stage-5):**

**Before:** `advisory_finding_key()` concatenated rule_id, resource_id, evidence_json with NUL-byte (`\x00`) separators.
**Vulnerability:** Evidence values containing literal NUL characters could collide (e.g., `rule=X, resource=Y, evidence=Z\x00abc` vs `rule=X, resource=Y\x00a, evidence=bc` both hash the same).
**Fix:** Switched to `sha256(json.dumps([rule_id, resource_id, evidence_json]))`. JSON serialization is lossless and guarantees unique canonical form.
**Consequence:** Bumped `Rule.evidence_key_version` field from implicit 0 to explicit 1, creating forward contract. Future rule evidence changes can increment the version independently; manifest documents the algorithm string `"sha256(..., version, ...)"` starting in v0.6.0.

This collision class is non-deterministic in practice (NUL characters are rare in Azure resource IDs and logs) but represents a protocol vulnerability that should be corrected upstream rather than papered over at the export layer.
